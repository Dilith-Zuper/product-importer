- `lib/account-catalog.js` — shared catalog-source abstraction for the option audit + backfill, mirroring `zuper-importer/lib/catalog-source.ts`. `audit-account-options.js` and `backfill-account-options.js` now take `--source srs|qxo|abc` (default `srs`, so SRS behavior is unchanged) and can remediate **QXO- and ABC-imported accounts**, where the account product carries a digit-only `product_id` derived from a text catalog key. The helper resolves it back to the catalog: SRS is identity (numeric PK); QXO digit-strips every `product_key` (`C-000SP5` → `5`); ABC digit-strips `abc_products.product_id` (`PFam_3272874` → `3272874`) — the *same* `Number(key.replace(/\D/g,''))` rule the wizard upload uses, so the link is exact. ABC mirrors SRS columns (`color_name`/`size_name`/`is_restricted`) so it gets color+size+composite; QXO is **color-only** (no single size column; wizard uploads QXO `includeSize=false`), so size/composite auto-disable under `--source qxo`. Key→id collisions are flagged (validated zero across all 76,812 QXO and 34,868 ABC products). Run e.g. `node audit-account-options.js --key XXX --login <name> --source abc --label "Acct"` then `node backfill-account-options.js … --source abc --test-one|--dry-run|--run`.

- `--source qxo-sku` — a **second QXO key scheme**. Elite Options Contracting turned out to be mixed-source: besides the 934 ABC products, 111 were QXO loaded with a single `qxo_variants.variant_sku` stamped as the Zuper `product_id` (e.g. `656024` → parent product `C-635001`), NOT the wizard's digit-stripped `product_key`. `lib/account-catalog.js` gained a `keyMode` (`identity` / `pk-digitstrip` / `variant-sku`): `qxo-sku` resolves the variant's parent `product_key` and loads ALL the parent's colors onto the product, fanning a parent's option set out to every account product that points at it. Color-only. Run on Elite Options: 73 should-have-color, 3 present, **70 missing** → backfilled 70/70, 0 failures (1 product truncated from 57→50 at Zuper's option cap); re-audit confirmed 73/73 present, 0 missing.

- `update-vendor-catalog.js` — **reusable vendor-catalog SKU writer**. Populates a Zuper vendor's catalog with one `vendor_sku` per product from the source DB (`lib/account-catalog.js` `resolveOwnVariantSkus`). Built for variant-sku-keyed sources (`--source qxo-sku`), where the Zuper `product_id` IS a specific `variant_sku`, so each product maps to exactly one orderable SKU (`--sku-field variant_sku|product_number|manufacturer_number`, default `variant_sku`); `vendor_cost` = the product's purchase price. `--dry-run`/`--run`, `--vendor "<name>"` (default per source). Reads the existing catalog via `GET /vendor_catalogs?filter.vendor` and skips products/SKUs already present (idempotent). Documents the Zuper vendor-catalog quirks it had to work around (see below).
  - **Zuper vendor-catalog API gotchas (discovered here):** entry schema is `{ product, vendor_sku, vendor_cost, remarks }` — `vendor_cost` MUST be a string and `remarks` is required; a numeric cost is rejected with a *misleading* `"Vendor SKU should be unique for each product in the catalog"` 400. The batch POST to `/vendors/{uid}/catalog` is **non-atomic** (persists rows before erroring). `GET /vendors/{uid}` does NOT return the catalog (its `vendor_catalog` field is always empty) — list via `GET /vendor_catalogs?filter.vendor={uid}`; entries carry `vendor_catalog_uid` (delete/update via `/vendors/{uid}/catalog/{catalog_uid}`). There is **no per-color SKU** — each entry's `options[]` holds only `option_uid`s and is derived **live** from the product's current `option_values` (so the option backfill automatically flows through to the catalog). `vendor_sku` must be unique across the whole catalog.
  - Elite Options Contracting (QXO vendor): catalog was already populated 2026-06-17 (115 entries, `vendor_sku` = each product's `variant_sku`, options live-reflecting the colors we backfilled today). Net change this session: removed one stray test entry; no rewrite needed.

### Changed
- `backfill-account-options.js` + `audit-account-options.js` — now load **SIZE** options too, not just colors (driven by the High Impact account, which had impact-rated VELUX skylights uploaded with no options because they vary by size, not color). The option axis is chosen per product from `srs_variants`: both color & size vary → **composite** values (`Black — 36" x 144'`) built from the *real* variant combinations (not a cartesian product); only color varies → color; only size varies → size; constant-color products with varying sizes load the sizes (don't collapse to the one color). Size/composite are non-mandatory tags; SHINGLES color stays mandatory. New `--colors-only` reverts to the legacy color-only behavior, and `--upgrade` also re-writes already-optioned products when the catalog now implies a different set (e.g. a color-only product that should become composite once sizes are added; no-ops when the live set already matches). The audit's buckets (MISSED color/size/composite, all "fixable") mirror exactly what the backfill will write. Option values still capped at 50 (see note).

### Added
- `export-zuper-products.js` — **reusable READ-ONLY Zuper catalog export** for accounts NOT imported by the SRS tool (text/manual `product_id`s, where `audit-account-options.js` finds 0 numeric SRS parts). Pages all products via the same flexible connection (`--login`/`--region`/`--base`, `--key`/`ZUPER_API_KEY`) and `product`/`products` auto-resolution as the audit/backfill tools, then writes a `<label>-zuper-products.xlsx` with headers the name-based matcher auto-detects (`Product Name`, `Product Description` from `plain_text_description`, `Brand` from the top-level `brand` field, `UOM`, `Price`) plus `Zuper product_id`/`Type`/`Category` reference columns. `--parts-only` skips SERVICE/BUNDLE/PRODUCT rows. Bridges the gap so a non-SRS account can be run through `match-account-to-srs.js`. First run (STX Roofing, `us-east-1`): 652 products → matcher reported 59 exact / 87 strong / 160 weak / 260 none, 306/566 material parts in SRS.
- `audit-account-options.js` — **reusable READ-ONLY audit** (pairs with the backfill below): for any SRS-imported Zuper account, reports which numeric-SRS products should have color options (per `srs_variants`) but have an empty option block in Zuper, plus size-only parts. Same flexible connection (`--login`/`--region`/`--base`, `--key`/`ZUPER_API_KEY`) and `product`/`products` endpoint auto-resolution as the backfill. Writes a label-prefixed Excel report (`<label>-options-audit.xlsx`) with the missing-option products: SRS id, name, category, brand, expected option values, Zuper `option_values` count, and `product_uid`. High Impact Roofing post-backfill: 4,404 should-have-color / 4,404 present / 0 missed.
- `backfill-account-options.js` — **reusable color-option backfill** for any Zuper account imported from SRS (generalized from the one-off High Impact Roofing remediation). Finds products that have colors in `srs_variants` but were uploaded with an empty `option` block (the footprint of the variant-pagination bug fixed in zuper-importer) and repopulates `option.option_values` via GET → PUT — round-tripping the whole product so only `option` changes. Connection is flexible: `--login <name>` (resolves region via `accounts.zuperpro.com/api/config`), `--region <region>`, or `--base <url>`; the API key comes from `--key` or `ZUPER_API_KEY` (no longer hardcoded). Auto-resolves the `product`-vs-`products` list endpoint per data-center. Flags match the wizard (`SHINGLES` → customer_selection/mandate true, else false; `--selection all` to make every color product selectable). Safe rollout via `--test-one` / `--dry-run [--limit N]` / `--run [--limit N]`; writes a per-product results JSON prefixed by the account label. First real run (High Impact Roofing, `us-west-1c`): 2,947 products backfilled, 0 failures, re-scan confirmed 0 remaining. Read-only against Supabase; no SKUs (those are the separate vendor catalog).
- `match-account-to-srs.js` — generalized the A&A↔SRS matcher into a **reusable, account-agnostic tool**. Takes any Zuper "Product Report" export and matches it against the SRS catalog: `node match-account-to-srs.js <input.xlsx> [--label "Name"] [--out "Prefix"] [--sheet "Sheet"] [--name-col/--brand-col/… overrides]`. Columns are auto-detected from header aliases (`Product Name`/`Name`/`Item`, `Brand`/`Manufacturer`, `UOM`/`Unit`, `Price`/`Sell Price`, `#`/`No`) with a substring fallback; only the name column is required (errors listing the seen headers otherwise). Output filenames derive from `--out` (`<OUT>_SRS_Match.xlsx` + `… - simple.xlsx` + `<OUT>_SRS_Review.xlsx`) and the account label prefixes the output column headers. Exposes `run({ inFile, label, outPrefix, sheet, cols })` for programmatic use. All matching logic (type-gating, distinguishers, brand gating, material/labor split, options rollup, 3 output sheets) is unchanged — just parameterized. Verified on the 663-row `hackthon 4 product export.xlsx` (auto-detected all columns; 619 exact / 27 strong / 17 weak / 0 none).
- `match-aa-to-srs.js` is now a **thin wrapper** that calls `match-account-to-srs.js` with the A&A input + `A&A` label/prefix, so the original command and its `A&A_SRS_*` output names are unchanged (re-verified identical: 29 exact / 39 strong / 21 weak / 14 none).
- `match-aa-to-srs.js` (original) — adapts the QXO matcher to answer "which A&A material parts exist in SRS, and do we have options for them" for A&A's 141 free-text Zuper parts (`aanda_parts_export.xlsx`). Splits the list into material vs non-material first: labor/service/fee/change-order rows (Install…, Remove…, Dump Fee, Freight, Travel, Deposit, `Contract to Match …`, `CO - … Replacement`, Change Order) are detected by **name** (not description — A&A descriptions are marketing copy that routinely say a part is "installed …") and routed to a separate "Excluded (labor & fees)" sheet, never matched. The 103 material parts are scored against all 19,807 `srs_products` (token overlap + dimension signals). Brand is detected from the part name (incl. CT→CertainTeed alias + Norwesco/Versico/Pabco/Quarrix); when SRS carries that brand the scorer strongly prefers same-brand products (+0.30) and penalizes cross-brand (−0.25), so e.g. every `Versico …` / `Pabco …` / `Quarrix …` part resolves within its own manufacturer. Output reports detected brand, whether SRS carries it, confidence (exact/strong/weak/none), and for matches the unrestricted `srs_variants` rollup: `# Variants`, `Has options?`, colors/sizes, `product_options`, sample SKUs. A `SYNONYMS` map rewrites A&A shorthand before matching: `Dens Deck` → SRS's Elevate-branded `DensDeck … Roof Board`, and the `Tri-Built` house-brand label (SRS Distribution's own private label, not a manufacturer in our catalog) is stripped so those resolve to the generic synthetic-underlayment / ice-&-water equivalent. Writes `A&A_SRS_Match.xlsx` (2 sheets: "Material matches" + "Excluded (labor & fees)") plus a 6-column `A&A_SRS_Match - simple.xlsx` (A&A name, confidence, SRS id/name/category, options) and `A&A_SRS_Review.xlsx` (the 35 weak+none rows with top-3 SRS candidates each + blank "Correct SRS ID"/"Notes" columns for manual triage); all fall back to a timestamped sibling if the target is open in Excel (EBUSY). **Type-gating overhaul** (after a manual review caught ~20 brand-only/wrong-category matches — e.g. Versico TPO membrane → TPO *primer*, OSB → *Staples*, Roof Runner → CertaSeam *tape*): each name is tagged with coarse product type(s) via `TYPE_SIGS` (shingle/starter/ridge/vent/siding/underlayment/ice_water/flashing/membrane/primer_adh/fastener/tool/decking/insulation); confidently-typed pairs that share no type are penalized −0.40, same-type +0.12. A `DISTINGUISHERS` set penalizes form-defining tokens present on only one side (primer/cleaner/gun/staple/plate/rhinobond/kit/…) −0.16 each. The brand boost is now gated on a shared non-brand content token (brand alone no longer carries a wrong-type match), numeric/dimension weight cut 0.10→0.06, and a `roof runner → roofrunner` synonym added. Run: 29 exact / 39 strong / 21 weak / 14 none → 89/103 material parts matched (68 high-confidence vs 57 pre-overhaul), 88 with options; 38 excluded as labor/fees. Brand audit beyond the 6 named (Certainteed 495, Gaf 522, Norwesco 60, Versico 250, Pabco 12, Quarrix 12): also present — Apoc (84), Chem Link (31), Lp (62); absent as a brand — Tri-Built (SRS private label, 0), and metal panels Tuff-Rib / Snaplock (SRS thin on standing-seam panels).
- `srs-add-order-uom-column.sql` / `apply-srs-order-uom-column.js` / `enrich-order-uom.js` — new `srs_products.order_uom` column populated with the dominant `order_uom` (mode) across each product's unrestricted `srs_variants`, falling back to `product_uom[0]` for the 27 products with no variant UOM data. All 19,807 products populated.

### Fixed
- SRS `product_uom[0]` (the value zuper-importer's `toZuperUom()` used as "the" Zuper UOM) is the raw source array's first element, which is frequently `PAL` (pallet — a near-universal secondary/bulk unit on almost every variant) rather than the actual order unit. Verified 11,236/19,807 products (57%) had `product_uom[0]` ≠ the dominant variant `order_uom`, and 7,533 (38%) would upload with the wrong Zuper UOM (e.g. `EA` instead of `PC`/`RL`/`BX`/`BDL`). Same class of bug fixed for ABC in `9c1e6a2`; fix is the new `order_uom` column above — zuper-importer now reads it.
- ABC Big 3 tier rules brought to SRS parity in `enrich-abc-product-line-and-tier.py` (found in 2026-06-10 product review): CertainTeed Landmark PRO better→**good** and base Landmark good→**addon** (SRS canon: PRO is the current standard, base is the below-standard entry); OC base Duration better→**good** and Oakridge good→**addon**; GAF split "timberline hd" into HDZ→**good** / prior-gen HD→**addon** and Natural Shadow good→**addon**. Before this, ABC CertainTeed Good packages led with the wrong shingle vs the same brand on SRS. Requires re-running the script + matview refresh.

### Changed
- `lib/utils.js` `fetchAll()` now accepts an `orderBy` option. Without an explicit order, PostgREST pagination is not stable across `.range()` calls — same row can appear on adjacent pages or be skipped. All enrichment scripts that depend on full-table coverage should now pass `orderBy: '<pk_col>'`.
